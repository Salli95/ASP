# Task Management System - Midterm Assignment

## Теоретическое обоснование (Block 3)

Для интеграции с `NotificationService` (отправка Email при завершении задачи) я выбираю **Асинхронный паттерн интеграции** (Asynchronous Messaging).

### Почему выбран этот паттерн:
1. **Слабая связанность (Loose Coupling):** Сервис задач не должен знать ничего о работе сервиса уведомлений. Его задача — просто сообщить факте завершения задачи.
2. **Отказоустойчивость (Resilience):** Если сервис отправки почты временно недоступен, сообщение сохранится в очереди (например, в RabbitMQ) и будет обработано позже. Основной API при этом не выдаст ошибку пользователю.
3. **Производительность:** Отправка Email — медленная операция. Асинхронный подход позволяет API мгновенно отвечать пользователю, не дожидаясь завершения отправки письма.

### Технологии:
Для реализации будет использован брокер сообщений **RabbitMQ**. 
- **Издатель (Publisher):** Наш Task Service, который при срабатывании события `OnTaskCompleted` отправляет сообщение в очередь.
- **Подписчик (Subscriber):** Notification Service, который слушает очередь и обрабатывает отправку писем.
